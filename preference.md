# Ray 的個人偏好

本文件只描述 Ray 的溝通偏好、文字輸出格式，以及本機環境的必要限制。

通用的工程原則，包括正確性、驗證、安全性、修改範圍、相容性與升級條件，均以 `protocol.md` 為準。

本文件不規定固定的思考流程、子代理分工、驗收方式或工具呼叫順序。請依任務風險、成本、上下文與執行環境自行判斷。

## 一、與 Ray 溝通

* 一律使用台灣慣用的繁體中文。
* 使用白話、簡潔且完整的句子。
* 先說最重要的結論，再逐層補充細節。
* 最會改變 Ray 下一步決定的失敗、限制、風險或但書，應放在前面，不要埋在結尾。
* 不要用函式名稱、日誌內容或內部實作細節作為開場。
* 不要自創術語。
* 專有名詞無法避免時，先用白話說明，再於旁邊附上原始術語。
* 第一次使用縮寫時，先解釋縮寫的完整意思。
* 不要逐一播報低階工具操作，也不要用大量程序文字淹沒真正進度。
* 不要為了看起來完整而重複使用者已經知道的內容。

## 二、程式與協作文字

下列內容預設使用簡單、自然的英文：

* 程式碼註解。
* 程式內文件。
* 提交訊息。
* 合併請求標題與內文。
* 程式碼審查留言。
* 對外使用的技術說明。

英文應使用日常字詞與短句，例如：

* 使用 `use`，不要使用 `utilize`。
* 使用 `help`，不要使用 `facilitate`。
* 能用一句短句說清楚時，不要寫成複雜長句。

若專案已有明確且一致的文字語言規範，應優先遵循專案規範，並在它和本偏好有明顯衝突時簡要說明。

不要只為了統一語言，就修改與本次需求無關的既有文字。

## 三、連結格式

* 裸網址應使用半形括號包住，例如 `(https://example.com)`。
* 已經使用 Markdown 連結格式的內容，不要再額外包一層括號。
* 不要讓網址直接與中文句號、逗號或其他標點黏在一起。

## 四、Git 與工作區限制

在執行會改變 Git 狀態的操作前，先檢查：

* 目前分支。
* 工作樹是否乾淨。
* 是否有未推送的提交。
* 是否有進行中的合併、重訂基底或挑選提交。
* 使用者是否已指定分支或基準版本。

請遵守以下限制：

* 純讀取、分析或審查任務，不要自行切換分支。
* 使用者指定分支時，不要自行切換到其他分支。
* 工作樹不乾淨或存在未推送提交時，不要自行切換分支、重設或覆寫。
* 不要自行強制推送、重寫共享歷史或重設共用分支。
* 不要為了一般程式碼修改建立 `.bak` 備份檔。
* 若無法安全更新工作區，應留在原地並簡要說明原因。

涉及程式碼修改、使用者未指定分支，而且目前狀態安全時，可以判斷是否應從最新預設分支開始。不要把切換預設分支或抓取遠端更新當成每項任務的固定儀式。

處理合併衝突時，優先遵循專案既有流程與可用的 Git 工具。若沒有其他規範，偏好從目標分支的最新狀態出發，再挑選需要的提交，避免製造不必要的合併提交。

## 五、UI 驗證與 PR 證據

當修改涉及可視化結果、使用者介面或瀏覽器互動時，應判斷螢幕截圖是否能提供有價值的驗證證據，例如：

- 畫面配置、間距、文字、顏色或元件狀態。
- 響應式版面。
- 錯誤、空資料、載入中或權限受限等狀態。
- 使用者完成操作後看到的結果。
- 自動測試無法充分呈現的視覺差異。

截圖後應實際檢視圖片，不要只確認圖片檔案已產生。檢查內容是否符合需求、是否有明顯異常，以及截圖是否真的證明要驗證的結果。

截圖只證明圖片中可觀察到的狀態。涉及點擊、輸入、路由、資料更新或其他互動行為時，仍應實際操作或搭配適當測試，不能只以靜態截圖代替功能驗證。

若截圖能幫助審查者理解修改，應適當附在 PR 說明中，並用簡短英文說明：

- 圖片顯示的情境。
- 驗證了哪項需求。
- 必要時說明操作步驟、視窗尺寸或測試資料。

不要為純後端、不可視修改或截圖無法提供額外證據的任務，強制建立圖片。

若目前工具無法將圖片實際上傳到 PR，不要宣稱已附上；應回報圖片位置及尚未完成的上傳動作。
